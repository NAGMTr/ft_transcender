# Friend List data model and flow

This document explains the Friend List (friendship + friend requests) data model and request flow, based on app-backend/src and app-frontend/src.

## Overview

There are two separate concepts:

1. Friend requests (pending/accepted/declined)
2. Confirmed friendships (the friends list)

Friend requests are stored in the FriendRequest entity. Confirmed friendships are stored in a join table created by the User entity's ManyToMany relation.

## Data model (tables and relations)

### users

- Table name: users
- Columns (relevant): id, username, email, avatar_url, is_online, created_at, update_at
- Relationships:
  - friends: ManyToMany self-relationship through the join table user_friends
  - sentRequests: OneToMany to FriendRequest (sender)
  - receivedRequests: OneToMany to FriendRequest (receiver)

Why it is structured like this:

- Users are the core entity.
- The friends list is a ManyToMany self-relationship because a user can have many friends and each friend is also a user.
- Friend requests are modeled as a separate entity to keep request history/status and allow pending/declined states without immediately adding to the friends list.

### friend_request (entity name: FriendRequest)

- Table name: generated by TypeORM from the entity name (default naming depends on configuration)
- Columns:
  - id (PK)
  - sender (FK to users)
  - receiver (FK to users)
  - status (enum: PENDING, ACCEPTED, DECLINED)
  - created_at

Why it is structured like this:

- Keeps a record of who sent the request to whom and its status.
- Prevents adding users to the friends list before acceptance.
- Supports query for pending requests for a given receiver.

### user_friends (join table)

- Table name: user_friends
- Columns:
  - user_id (FK to users.id)
  - friend_id (FK to users.id)

Why it is structured like this:

- Represents confirmed friendships as a ManyToMany mapping.
- Decouples friendship from the request workflow so you can have accepted friendships without storing extra fields on the join table.

Important behavior note:

- The current addFriend implementation only inserts one direction (user_id -> friend_id). If you want friendships to be symmetric (both users see each other), you need to add the reverse relation too.

## Backend flow (friend request)

### Send request

Endpoint:

- POST /friend-requests

Service logic:

1. Find sender by senderId.
2. Find receiver by id or username.
3. Block self-requests.
4. Check if a request already exists in either direction.
5. Create a new FriendRequest with status PENDING.

### List pending requests

Endpoint:

- GET /friend-requests/pending/:userId

Service logic:

- Finds FriendRequest entries where receiver.id = userId and status = PENDING.
- Loads the sender relation so the UI can show sender.username.

### Accept request

Endpoint:

- PATCH /friend-requests/:id/accept

Service logic:

- Loads the request and sets status = ACCEPTED.
- It does NOT create a row in user_friends.

### Reject request

Endpoint:

- DELETE /friend-requests/:id/reject

Service logic:

- Deletes the FriendRequest row entirely.

## Frontend flow (FriendsList)

Component:

- app-frontend/src/components/FriendsList.tsx

1. On load, it fetches current friends with GET /api/users/:userId/friends.
2. It fetches pending requests with GET /api/friend-requests/pending/:userId.
3. To send a request, it calls POST /api/friend-requests with senderId and receiverId (string or number).
4. To accept a request, it calls PATCH /api/friend-requests/:id/accept.
5. To reject a request, it calls DELETE /api/friend-requests/:id/reject.
6. To remove a friend, it calls DELETE /api/users/:userId/friends/:friendUsername.

## Why it is built this way

- The friend_request table handles the workflow of pending/accepted/declined without modifying the friends list prematurely.
- The user_friends join table represents only confirmed friendships.
- This separation keeps the friends list query fast and clean (it only needs to load confirmed friends).

## Current gaps / inconsistencies to be aware of

- Accepting a request only changes status to ACCEPTED; it does not add a friendship to user_friends. If you expect an accepted request to show up in the friends list, a follow-up step is needed (either in acceptRequest or in the frontend after acceptance).
- The remove-friend endpoint in the backend expects a username for the user path param. The frontend currently sends userId in that position, which will not match the backend lookup.

## Visual summary (data structure)

### Relation map (table)

| Entity / Table | Key columns                                                       | Relation                              | Notes                                        |
| -------------- | ----------------------------------------------------------------- | ------------------------------------- | -------------------------------------------- |
| users          | id, username, email, avatar_url, is_online, created_at, update_at | Self ManyToMany via user_friends      | Confirmed friends only                       |
| friend_request | id, sender_id, receiver_id, status, created_at                    | ManyToOne to users (sender, receiver) | Pending/accepted/declined workflow           |
| user_friends   | user_id, friend_id                                                | Join table for users.friends          | Current implementation inserts one direction |

### Data flow (tree)

```
users
|-- id (PK)
|-- username
|-- email
|-- avatar_url
|-- is_online
|-- created_at
|-- update_at
|-- friends (ManyToMany)
|   `-- user_friends
|       |-- user_id (FK -> users.id)
|       `-- friend_id (FK -> users.id)
|-- sentRequests (OneToMany)
|   `-- friend_request
|       |-- id (PK)
|       |-- sender_id (FK -> users.id)
|       |-- receiver_id (FK -> users.id)
|       |-- status (PENDING | ACCEPTED | DECLINED)
|       `-- created_at
`-- receivedRequests (OneToMany)
  `-- friend_request (same table, receiver_id points to users.id)
```

## Visual summary (accept / reject outcome)

```
Pending request (friend_request.status = PENDING)
        |
        |-- Accept (PATCH /friend-requests/:id/accept)
        |     |
        |     |-- Update friend_request.status -> ACCEPTED
        |     `-- (No row added to user_friends by default)
        |
        `-- Reject (DELETE /friend-requests/:id/reject)
              |
              `-- Delete row from friend_request
```
